# 시스템 구조 개요 (Architecture Overview)

본 문서는 히터 캘리브레이션 소프트웨어의 **개념적인 시스템 구조**와  
각 구성 요소의 역할을 설명하기 위한 문서입니다.

빌드 가능한 코드 전문이 아니며,
포트폴리오 용도로 설계 방식과 구조적인 접근 방법을 정리한 내용입니다.

---

## 전체 구조 개요

시스템은 크게 다음과 같은 계층으로 구성됩니다.

- 통신 계층 (Communication Layer)
- 애플리케이션 계층 (Application Layer)
- 유틸리티 계층 (Utility Layer)

각 계층은 역할에 따라 분리되어 있으며,  
서로 간의 의존성을 최소화하는 것을 목표로 설계되었습니다.

---

## 1. 통신 계층 (Communication Layer)

### 역할
- 외부 장비와의 시리얼 통신 처리
- 통신 채널과 프로토콜 로직 분리
- 장비 종류 또는 통신 방식 변경에 대비한 확장성 확보
- DB 서버 연동 및 통신 데이터 처리
- FTP 서버 연동 및 통신 데이터 처리

### 설계 포인트
- 통신 채널(Serial 등)을 추상화하여 상위 로직에서 I/O 세부사항을 알 필요가 없도록 구성
- 프로토콜 파싱 로직을 별도 클래스로 분리하여 재사용 가능하도록 설계
- 통신 오류 및 예외 상황을 상위 계층으로 전달할 수 있도록 구조화

---

## 2. 애플리케이션 계층 (Application Layer)

### 역할
- 테스트 시퀀스 및 작업 흐름 제어
- 캘리브레이션 진행 상태 관리
- 측정 결과에 대한 판정 로직 처리 (PASS / FAIL)

### 설계 포인트
- 테스트 흐름을 중앙에서 제어하여 UI 또는 통신 로직과 분리
- 판정 기준을 명확히 분리하여 유지보수 및 기준 변경에 용이하도록 구성
- UI 이벤트에 직접 종속되지 않도록 계층 간 의존성 최소화

---

## 3. 유틸리티 계층 (Utility Layer)

### 역할
- 테스트 로그 기록
- 측정 결과 데이터 저장 및 CSV 출력
- 공통 기능 제공

### 설계 포인트
- 로깅 및 데이터 저장 로직을 독립된 모듈로 구성
- 테스트 결과 추적성과 이력 관리에 중점을 둔 구조
- 추후 데이터 분석이나 외부 시스템 연계를 고려한 형식 유지

---


## 구조 설계 의도 요약

본 시스템은 다음과 같은 설계 원칙을 기준으로 구성되었습니다.

- 역할과 책임이 명확한 계층 분리
- 통신, 로직, UI 간 의존성 최소화
- 유지보수 및 확장성을 고려한 구조
- 테스트 흐름과 판정 로직의 명확한 분리

본 문서는 실제 구현을 모두 공개하는 것이 아닌,  
**개발자의 설계 관점과 코드 구성 방식을 이해하기 위한 참고 자료**입니다.
